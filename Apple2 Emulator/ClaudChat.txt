Deciphered emulator screen rendering mechanisms for retro systems
This is where things get more nuanced. Basic 6502 emulators like py65 don't handle the Apple II screen at all — they're CPU/memory emulators only, with no video subsystem. To emulate the Apple II display, you need a fuller system emulator built on top of a 6502 core.

Apple II Video Complexity
The Apple II screen is famously tricky to emulate because it's not a simple framebuffer. It has several distinct video modes:

Text mode – 40 columns × 24 rows of characters, stored in specific memory pages ($0400–$07FF and $0800–$0BFF for page 2)
Lo-res graphics – 40×48 blocks of 16 colors, sharing the text memory pages
Hi-res graphics – 280×192 pixels, stored at $2000–$3FFF (page 1) or $4000–$5FFF (page 2), with a very quirky color encoding
Mixed mode – top portion hi-res or lo-res, bottom 4 lines text
The hi-res mode is the hardest to emulate correctly. Its color system is artifact-based — it exploits the NTSC color signal. Each byte controls 7 pixels, and a high bit shifts the color phase, producing colors (green, purple, blue, orange, white, black) that depend on the pixel's horizontal position and neighboring pixels. On a real monitor it's an analog trick, not a digital palette lookup.

What Python Apple II Emulators Do
A few Python projects attempt fuller Apple II emulation:

PyApple2 / apple2py (various GitHub projects) These typically render the screen using pygame or tkinter, implementing:

A character ROM lookup table for text mode
A color lookup table for lo-res graphics
An NTSC artifact color algorithm (or a simplified approximation) for hi-res
The typical hi-res approach in Python emulators:

python
# Simplified — real implementations are more involved
COLORS = {
    (0, 0): BLACK,
    (1, 0): GREEN,   # odd pixel, phase 0
    (1, 1): ORANGE,  # odd pixel, phase 1
    (0, 0): BLACK,
    ...
}
# For each byte in $2000-$3FFF, extract 7 bits,
# check the high "palette" bit, then determine
# color based on bit position (odd/even) and neighbors
linapple / AppleWin are not Python but are the reference for "correct" color behavior if you want to validate an implementation.

Practical Reality
Most Python-based attempts are partial or approximate:

Text mode is straightforward and usually well-implemented
Lo-res is also manageable
Hi-res color is often approximated rather than perfectly emulated, especially the NTSC artifact fringing between pixels
Almost none emulate the mixed-mode timing edge cases or the exact scanline behavior of the original hardware
If you're building something in Python and need accurate Apple II video, the usual approach is to use pygame for the display surface, implement the memory-mapped regions ($0400 for text, $2000 for hi-res), and write a rendering function that maps those memory regions to screen pixels on each frame. Getting hi-res color fully right is a rabbit hole — many emulators just use a pre-baked lookup table of the 6 main colors based on bit position and the palette bit.

Can I legally use apple2 bios?

Weighed legal considerations regarding Apple II BIOS usage
Weighed legal considerations regarding Apple II BIOS usage
This is a nuanced area. I'm not a lawyer, so take this as general information rather than legal advice.

The Short Answer
No, not without permission — the Apple II ROMs are still under copyright, owned by Apple Inc. Using them without a license is technically copyright infringement.

The Longer Picture
What the Apple II ROMs contain:

The Apple II BIOS/firmware includes the Monitor (the low-level machine language interface), Applesoft BASIC (which was actually licensed from Microsoft), the disk controller routines, and the character ROM (the font). Each of these is a separate copyrightable work.

Apple has never officially released the ROMs as free or open source. Unlike some old companies that have abandoned their IP or explicitly granted permission, Apple actively maintains its copyright portfolio and has never stated that Apple II ROMs are free to use.

The "abandonware" argument doesn't hold legally. Many people treat old software as fair game because it's no longer sold, but copyright doesn't expire just because a product is discontinued. In the US, copyright on software from the early 1980s won't expire for decades yet.

What Emulator Projects Actually Do
Legitimate emulator projects handle this a few ways:

Require the user to dump their own ROMs from real hardware they own. This is the most common approach — the emulator ships without the ROM, and you supply it yourself. Personal use of a ROM dump from hardware you own sits in a legal grey area that has generally not been prosecuted.

Open source ROM replacements — projects like Applewin and others can work with the original ROMs but don't distribute them. The community has also produced open source replacements for some ROM functionality, though Applesoft BASIC (the Microsoft-licensed part) makes a fully clean-room replacement complicated.

CFFA / open hardware projects — the broader Apple II retrocomputing community has produced open source hardware and firmware that is compatible but not derived from Apple's originals.

The Practical Reality
In practice, Apple has not aggressively pursued emulator authors or individuals who dump ROMs from their own hardware. The retro computing community operates largely in a tolerated grey zone. But "Apple hasn't sued anyone recently" is very different from "it's legal."

For a serious project you have a few clean options: require users to supply their own ROM dump, write a clean-room replacement (hard but doable for the Monitor; complicated for Applesoft due to the Microsoft license), or reach out to Apple — some companies have granted permission for non-commercial preservation/emulation use when asked directly, though Apple has generally not been responsive to such requests historically.

The practical implication: With Microsoft having made the BASIC source code available, someone could potentially put together a ROM image not containing any code for which Apple owns the copyright — and substitute the Apple-specific parts with new clean-room code, similar to what some Taiwanese Apple II clone manufacturers did in the 1980s.

So the bottom line is: Microsoft's core 6502 BASIC is now legally free and open source, which is a big deal for the retro computing community. But a complete, legally clean Applesoft BASIC — including all the Apple II graphics commands — still requires either Apple's blessing or someone writing clean replacements for the Apple-specific extensions. That's a really smart approach — it's called High Level Emulation (HLE) as opposed to Low Level Emulation (LLE). Instead of faithfully emulating every ROM byte, you intercept calls at known entry points and substitute your own modern implementations. This is exactly how many successful emulators work.

The Core Idea
The Apple II has well-documented ROM entry points and DOS/ProDOS vectors. When your 6502 CPU emulator is about to JSR or JMP to a known address, you intercept it and run Python code instead:

python
KNOWN_VECTORS = {
    0xFDED: handle_cout,      # Character output (COUT)
    0xFD0C: handle_cin,       # Character input (RDCHAR)
    0xFB1E: handle_init,      # Hardware init - no-op in emulator
    # DOS 3.3 entry points
    0x03D9: handle_dos_call,  # Main DOS entry point
}

def cpu_step(cpu, memory):
    pc = cpu.PC
    if pc in KNOWN_VECTORS:
        KNOWN_VECTORS[pc](cpu, memory)  # run Python instead of ROM
        cpu.PC = memory.pop_return()    # simulate RTS
    else:
        cpu.execute_instruction()
File System Interception
This is where it gets powerful. DOS 3.3 and ProDOS use well-known entry points and calling conventions:

python
# DOS 3.3 file command structure at $200 (command buffer)
DOS_COMMANDS = {
    0x01: dos_load,
    0x02: dos_save,
    0x04: dos_read,
    0x08: dos_write,
    0x10: dos_delete,
    0x20: dos_catalog,  # LIST files = ls on Linux folder
}

def dos_catalog(cpu, memory):
    """Instead of reading disk sectors, list a local Linux folder."""
    import os
    files = os.listdir(self.disk_path)
    output = "\nDISK VOLUME 254\n\n"
    for f in files:

    print_to_apple_screen(memory, output)

def dos_load(cpu, memory):
    """Instead of reading disk sectors, open a local file."""
    filename = read_apple_string(memory, 0x0200)  # filename from buffer
    filepath = os.path.join(self.disk_path, filename)
    with open(filepath, 'rb') as f:
        data = f.read()
    load_addr = read_word(memory, 0x0206)  # target address from call

Examined Apple II boot mechanics and disk format requirements
Great question, and it gets to the heart of a key design decision. Let me break it down:

What Actually Happens During Boot
When an Apple II powers on, the CPU starts executing ROM at $FFFC (reset vector), which jumps into the Monitor. The Monitor then reads one specific 256-byte sector — Track 0, Sector 0 — from the disk using a tiny bootstrap routine burned into the Disk II controller ROM at $C600. That sector contains a loader that pulls in more sectors, eventually loading DOS or ProDOS into RAM.

So yes, the very first moments of boot are deeply tied to the physical disk format — GCR encoding, soft-sectoring, track/sector layout — all of it.

The Three Approaches
Option 1: Use a standard .dsk or .nib disk image

The Apple II emulation community has solved this completely. Standard disk image formats are well documented:

.dsk — 140KB flat image, 35 tracks × 16 sectors × 256 bytes, straightforward to parse
.nib — raw nibble-level image, more accurate but more complex
.po — ProDOS-ordered variant of .dsk
You implement just enough low-level disk emulation to satisfy the boot loader, then hand off to your HLE layer once DOS is in RAM:

python
class DiskImage:
    def __init__(self, path):
        with open(path, 'rb') as f:
            self.data = f.read()  # 143,360 bytes for a .dsk
    
    def read_sector(self, track, sector):
        # DOS 3.3 sector skew table
        SKEW = [0,7,14,6,13,5,12,4,11,3,10,2,9,1,8,15]
        offset = (track * 16 + SKEW[sector]) * 256
        return self.data[offset:offset+256]
The Disk II controller ROM at $C600 calls into your disk emulation, you serve it sectors from the .dsk file, boot completes normally, DOS lands in RAM, and from that point on your HLE intercepts take over.

Option 2: Skip the boot process entirely

This is actually viable for your use case. Instead of booting, you:

Pre-load DOS or ProDOS directly into the memory locations it would normally occupy after boot ($9D00 area for DOS 3.3, $BF00 for ProDOS)
Pre-load your binary or GraFORTH image at its expected address
Set the CPU program counter directly to the entry point
Start executing
python
def cold_start_with_program(memory, cpu, program_path):
    # Load a minimal DOS stub into expected RAM locations
    memory[0x9D00:0xBFFF] = minimal_dos_stub()
    
    # Load the actual program
    with open(program_path, 'rb') as f:
        data = f.read()
    load_addr = 0x0803  # typical Applesoft binary start
    memory[load_addr:load_addr+len(data)] = data
    
    # Jump straight to it
    cpu.PC = load_addr
```

No disk format needed at all. This works well if you know what you're loading and where it expects to be in memory.

**Option 3: Hybrid — boot from disk image, then HLE takes over**

This is probably the most pragmatic for your goals:
```
Boot phase:   .dsk file → real sector-level emulation → DOS loads into RAM
              (just ~1 second of emulated time, a few hundred disk reads)
                         ↓
Runtime phase: HLE intercepts all DOS calls → Linux filesystem
               HLE intercepts ROM calls → pygame / your code
```

You only need real disk emulation for the boot sequence. Once the `$03D9` DOS entry point is live in RAM, you patch it or intercept it and take over. The disk image is only touched during those first few seconds.

## What the Boot Sequence Actually Reads

For DOS 3.3, the complete boot reads these sectors in order:
```
Track 0, Sector 0  → bootstrap loader (loaded by Disk II ROM)
Track 0, Sectors 0-9 → rest of DOS loader  
Track 17, various  → VTOC and directory
Track 17, Sector 0 → HELLO program filename
Various tracks     → HELLO program itself (usually Applesoft)
For a GraFORTH disk specifically, it skips the BASIC HELLO program and boots straight from the Monitor, so the boot sequence is actually simpler — fewer sectors read before it hands control to GraFORTH's own startup code.

How the Disk II Controller Works
The Disk II controller ROM at $C600 communicates with drives through soft switches — memory-mapped I/O addresses in the $C0x0-$C0xF range where x is the slot number. For slot 6 that's $C0E0-$C0EF:

$C0E0  - Drive 1 off
$C0E1  - Drive 1 on       ← which drive is selected
$C0E2  - Drive 2 off
$C0E3  - Drive 2 on       ← switch here for drive 2
$C0E4  - Motor off
$C0E5  - Motor on
$C0E6  - Select stepper phase 0
...
$C0EC  - Read/write latch
$C0ED  - Load data latch
$C0EE  - Read mode
$C0EF  - Write mode
Your emulator watches writes to these addresses and tracks which drive is currently selected:

python
class DiskController:
    def __init__(self, drive1_dsk, drive2_folder):
        self.drive1 = DskDrive(drive1_dsk)
        self.drive2 = VirtualDrive(drive2_folder)
        self.active_drive = 1
        self.track = 0
        self.phase = 0
    
    def write_soft_switch(self, addr, value):
        offset = addr - 0xC0E0
        
        if offset == 0x01:
            self.active_drive = 1
        elif offset == 0x03:
            self.active_drive = 2
        elif offset in (0x06, 0x07, 0x08, 0x09):  # stepper phases
            self.step_track(offset - 0x06)
    
    def read_soft_switch(self, addr):
        offset = addr - 0xC0E0
        if offset == 0x0C:  # read latch
            return self.current_drive().read_byte()
    
    @property
    def current_drive(self):
        return self.drive1 if self.active_drive == 1 else self.drive2
Drive 1 — Real .dsk Emulation
This needs to faithfully serve sectors so the boot ROM is happy:

python
class DskDrive:
    # DOS 3.3 logical to physical sector skew
    SECTOR_SKEW = [0,7,14,6,13,5,12,4,11,3,10,2,9,1,8,15]
    
    def __init__(self, path):
        with open(path, 'rb') as f:
            self.data = bytearray(f.read())  # 143,360 bytes
        self.track = 0
        self.sector = 0
        self.byte_pos = 0
        self._encode_tracks()  # pre-encode to GCR nibbles
    
    def _encode_tracks(self):
        """Pre-encode all sectors to GCR nibbles the ROM can read."""
        self.nibbles = []
        for t in range(35):
            self.nibbles.append(encode_track(self.data, t))
    
    def read_byte(self):
        """Return next nibble from current track stream."""
        track_data = self.nibbles[self.track]
        byte = track_data[self.byte_pos % len(track_data)]
        self.byte_pos += 1
        return byte
    
    def step(self, new_track):
        self.track = max(0, min(34, new_track))
        self.byte_pos = 0
The GCR encoding is the tricky part — the Disk II boot ROM expects to read properly encoded nibbles, not raw bytes. There are well-documented Python implementations of the 6-and-2 GCR encoding used by DOS 3.3.

Drive 2 — Virtual Linux Folder Drive
This is where it gets interesting. You have two choices for how to intercept:

Option A: Intercept at the DOS level (cleaner)

Once GraFORTH boots from Drive 1 and DOS 3.3 is in RAM, patch the DOS entry point at $03D9 to vector to your HLE handler. When GraFORTH does a file operation on Drive 2, DOS figures out it needs Drive 2, calls your handler, and you serve it from Linux:

Which Option to Choose
Option A (DOS-level intercept) is much simpler to implement and perfectly suits your hybrid design. The only time Drive 2 needs to look like a real disk is if GraFORTH's boot process touches it — which it won't, since Drive 1 handles the boot. By the time anything touches Drive 2, DOS is already in RAM and you can intercept cleanly.

Option B is the "pure" approach where even a low-level disk utility would see Drive 2 as a valid disk, but it's probably overkill for your goals.

The Seam Between the Two Drives
The beautiful thing about your design is the seam is natural — DOS 3.3 already handles drive switching transparently. GraFORTH source files and binaries you're working on live in your Linux folder on Drive 2. The GraFORTH environment itself boots from the real .dsk on Drive 1. From GraFORTH's perspective it just sees two drives exactly as it would on real hardware.

The Intercept Point
DOS 3.3 stores the target drive in a known zero-page location. You intercept at the DOS entry point, check which drive is being addressed, and only divert Drive 2 calls to Linux:

python
DOS_ENTRY   = 0x03D9
DRIVE_ZP    = 0x0243  # current drive (1 or 2) DOS is targeting
COMMAND_ZP  = 0x0242  # DOS command code
FILENAME_ZP = 0x0200  # filename buffer

DOS_COMMANDS = {
    0x01: 'INIT',
    0x02: 'LOAD',
    0x04: 'SAVE',
    0x06: 'CHAIN',
    0x08: 'WRITE',
    0x10: 'READ',
    0x20: 'DELETE',
    0x22: 'CATALOG',
    0x40: 'RENAME',
    0x80: 'OPEN',
}

def intercept_dos(cpu, memory):
    command = memory[COMMAND_ZP]
    drive   = memory[DRIVE_ZP]
    
    if drive != 2:
        # Drive 1 — let real DOS handle it, don't intercept
        return False
    
    # Drive 2 — handle with Linux
    filename = read_dos_filename(memory)
    handler  = DRIVE2_HANDLERS.get(command)
    
    if handler:
        handler(cpu, memory, filename)
        return True  # intercepted
    
    return False

DRIVE2_HANDLERS = {
    0x02: drive2_load,
    0x04: drive2_save,
    0x20: drive2_delete,
    0x22: drive2_catalog,
    0x40: drive2_rename,
}
The Handlers
python
BASE_DIR = "/home/user/apple2files"  # your Linux folder

def drive2_catalog(cpu, memory, filename):
    files = os.listdir(BASE_DIR)
    lines = ["\r\nDISK VOLUME 254\r\n"]
    for f in files:
        size = os.path.getsize(os.path.join(BASE_DIR, f)) // 256
        lines.append(f" B {size:03d} {f.upper()}\r\n")
    for line in lines:
        print_to_screen(memory, line)
    cpu.A = 0  # no error

def drive2_load(cpu, memory, filename):
    path = os.path.join(BASE_DIR, filename)
    if not os.path.exists(path):
        cpu.A = 0x08  # DOS error: file not found
        return
    with open(path, 'rb') as f:
        data = f.read()
    load_addr = read_word(memory, 0x026C)  # address DOS would use
    memory[load_addr:load_addr + len(data)] = data
    cpu.A = 0  # no error

def drive2_save(cpu, memory, filename):
    path  = os.path.join(BASE_DIR, filename)
    start = read_word(memory, 0x026C)  # start address
    length = read_word(memory, 0x026E)  # length
    data  = bytes(memory[start:start + length])
    with open(path, 'wb') as f:
        f.write(data)
    cpu.A = 0  # no error

def drive2_delete(cpu, memory, filename):
    path = os.path.join(BASE_DIR, filename)
    if os.path.exists(path):
        os.remove(path)
    cpu.A = 0

def drive2_rename(cpu, memory, filename):
    # DOS 3.3 RENAME passes old,new in the filename buffer
    old, new = filename.split(',')
    os.rename(
        os.path.join(BASE_DIR, old.strip()),
        os.path.join(BASE_DIR, new.strip())
    )
    cpu.A = 0

The Real Issue
The honest answer depends on one question: how much does raw emulation speed matter to you?

The Apple II runs at 1.023 MHz. A Python interpreter executing a naive fetch-decode-execute loop can do roughly 1-5 million simple operations per second depending on the machine. So Python is borderline — it might just barely keep up with a 1MHz 6502, but you'll have no headroom for the GUI rendering on top of that without careful optimization.

Java's JIT would give you 10-50x more headroom on the same CPU core loop, making real-time emulation comfortable.

My Recommendation
If you want to get something working quickly and are comfortable with Python → Python, but plan from the start to write the CPU core as a C extension or use a library like py65 rather than a pure Python loop.

If you want long-term performance and maintainability → Java, specifically with JavaFX for the GUI since Swing is showing its age.

There's also a third option worth considering:

Python for everything except the CPU core, using a small C extension just for the 6502 execute loop. This gives you Python's flexibility for all the HLE logic, file handling, and GUI while getting near-native speed where it matters. Something like:

┌─────────────────────────────────┐
│  Python                         │
│  - HLE intercept handlers       │
│  - Drive 1/2 logic              │
│  - pygame GUI                   │
│  - Memory bus / soft switches   │
├─────────────────────────────────┤
│  C extension (ctypes/Cython)    │
│  - 6502 fetch/decode/execute    │
│  - Runs at native speed         │
└─────────────────────────────────┘
This is actually how most serious emulators written in scripting languages are structured. Given that your project is heavy on the HLE/interception logic rather than cycle-accurate hardware emulation, this hybrid approach suits your design well.

  final Uint8List memory;
  
  AppleIIScreen(this.memory);
  
  @override
  void paint(Canvas canvas, Size size) {
    final paint = Paint();
    
    // Read hi-res page from memory $2000-$3FFF
    for (int y = 0; y < 192; y++) {
      for (int x = 0; x < 40; x++) {
        int addr = 0x2000 + hiresAddr(x, y);
        int byte = memory[addr];
        
        // Decode 7 pixels from this byte

          paint.color = on ? Colors.white : Colors.black;
          canvas.drawRect(
            Rect.fromLTWH(
              (x * 7 + bit) * 2.0,  // 2x scale
              y * 2.0,
              2.0, 2.0
            ),
            paint
          );
        }
      }
    }
  }
  
  @override
  bool shouldRepaint(covariant CustomPainter old) => true;
}

The Emulator Loop
Flutter uses a Ticker (tied to the screen refresh) or an Isolate (separate thread) to run the CPU. For an emulator you want the CPU running independently of the UI frame rate:

dart
class EmulatorState extends State<EmulatorWidget> 
    with SingleTickerProviderStateMixin {
  
  late Ticker _ticker;
  final CPU6502 cpu = CPU6502();
  final Uint8List memory = Uint8List(65536);
  
  @override
  void initState() {
    super.initState();
    
    // Run ~17000 cycles per frame (1MHz / 60fps)
    _ticker = createTicker((elapsed) {
      for (int i = 0; i < 17000; i++) {
        cpu.step(memory);
      }
      setState(() {});  // triggers repaint of screen widget
    });
    _ticker.start();
  }
}
For better performance you'd move the CPU into a Dart Isolate (like a thread) and only push the memory framebuffer region to the UI thread for rendering.

class Assembler6502:
    def __init__(self):
        self.labels = {}
        self.output = bytearray()
        self.origin = 0x0800
        self.pc = self.origin
    
    def assemble_file(self, path):
        with open(path) as f:
            lines = f.readlines()
        
        # Two passes — first to collect labels, second to emit bytes
        self.pass1(lines)
        self.pass2(lines)
        return self.output
    
    def pass1(self, lines):
        """Collect all label addresses."""
        pc = self.origin
        for line in lines:
            line = self.strip_comment(line)
            if not line:
                continue
            if line.endswith(':'):
                self.labels[line[:-1]] = pc
            else:
                pc += self.instruction_size(line)
    
    def pass2(self, lines):
        """Emit bytes."""
        for line in lines:
            line = self.strip_comment(line)
            if not line or line.endswith(':'):
                continue
            self.emit_instruction(line)
    
    def emit_instruction(self, line):
        mnemonic, *args = line.split()
        operand = args[0] if args else ''
        
        opcode, mode = self.resolve(mnemonic, operand)
        self.output.append(opcode)
        
        if mode == 'immediate':
            self.output.append(self.parse_byte(operand[1:]))
        elif mode == 'absolute':
            addr = self.resolve_label(operand)
            self.output.append(addr & 0xFF)
            self.output.append((addr >> 8) & 0xFF)
        elif mode == 'relative':
            target = self.resolve_label(operand)
            offset = target - (self.pc + 2)
            self.output.append(offset & 0xFF)

Identified obscure assembler feature and assessed Python implementation support
No Python assembler I know of supports that specific syntax out of the box. It's a feature from Apple II era assemblers — it looks like the Merlin or S-C Assembler style of local labels. You'd need to add it yourself, but it's actually one of the more elegant things to implement.

What The Feature Does
asm
        LDX #$10
^1      LDA $2000,X    ; ^1 — reusable label
        BEQ >2         ; branch forward to next ^2
        STA $3000,X
        DEX
        BNE <1         ; branch backward to previous ^1
^2      RTS            ; ^2
^n defines a local label that can be reused anywhere
>n means first occurrence of ^n ahead of current PC
<n means first occurrence of ^n behind current PC
Global labels (MYSUB:) work normally alongside them
Why It's Elegant
It solves the classic problem of inventing unique label names for tiny loops. Without it you end up with LOOP1, LOOP2, LOOP1A etc. With it you just use ^1 everywhere and the assembler figures out which one you mean based on direction.

How to Implement It in Python
The key insight is that local labels aren't resolved during pass 1 like global labels. Instead you collect all occurrences with their addresses, then during pass 2 resolve >n and <n by searching that list:

python
class Assembler6502:
    def __init__(self):
        self.global_labels = {}   # name → address
        self.local_labels  = []   # list of (number, address) in order
        self.output = bytearray()
        self.pc = 0x0800
    
    # ─── Pass 1 ───────────────────────────────────────────────
    def pass1(self, lines):
        pc = self.pc
        for line in lines:
            line = self.strip_comment(line).strip()
            if not line:
                continue
            
            if line.startswith('^'):
                # Local label — record number and address
                number = int(line[1:])
                self.local_labels.append((number, pc))
            
            elif line.endswith(':'):
                # Global label
                self.global_labels[line[:-1]] = pc
            
            else:
                pc += self.instruction_size(line)
    
    # ─── Pass 2 ───────────────────────────────────────────────
    def pass2(self, lines):
        for line in lines:
            line = self.strip_comment(line).strip()
            if not line:
                continue
            if line.startswith('^') or line.endswith(':'):
                continue  # labels don't emit bytes
            self.emit_instruction(line)
    
    # ─── Local label resolution ────────────────────────────────
    def resolve_operand(self, operand):
        """Resolve >n (forward) or <n (backward) local label."""
        
        if operand.startswith('>'):
            number = int(operand[1:])
            return self.find_local(number, direction='forward')
        
        elif operand.startswith('<'):
            number = int(operand[1:])
            return self.find_local(number, direction='backward')
        
        elif operand in self.global_labels:
            return self.global_labels[operand]
        
        else:
            return self.parse_literal(operand)
    
    def find_local(self, number, direction):
        """Find nearest local label ^number in given direction."""
        
        if direction == 'forward':
            # First ^number with address > current PC
            candidates = [
                addr for (n, addr) in self.local_labels
                if n == number and addr > self.pc
            ]
            if not candidates:
                raise AssemblerError(
                    f"No forward label ^{number} found from ${self.pc:04X}"
                )
            return min(candidates)  # nearest ahead
        
        else:  # backward
            # Last ^number with address < current PC
            candidates = [
                addr for (n, addr) in self.local_labels
                if n == number and addr < self.pc
            ]
            if not candidates:
                raise AssemblerError(
                    f"No backward label ^{number} found from ${self.pc:04X}"
                )
            return max(candidates)  # nearest behind
Handling Branch Instructions
Branch instructions (BNE, BEQ, BCC, BCS, BPL, BMI, BVC, BVS) need the resolved address converted to a signed relative offset:

python
def emit_branch(self, opcode, operand):
    target = self.resolve_operand(operand)
    # +2 because offset is relative to next instruction
    offset = target - (self.pc + 2)
    
    if offset < -128 or offset > 127:
        raise AssemblerError(
            f"Branch out of range: {offset} bytes from ${self.pc:04X}"
        )
    
    self.output.append(opcode)
    self.output.append(offset & 0xFF)  # handles negative via two's complement
    self.pc += 2

asm
        LDX #$10       ; PC=$0800

        LDA $2000,X    ; PC=$0802

^2                     ; local_labels → [(1,$0802), (2,$080E)]

Search local_labels for n==2 and addr > $0805
Finds $080E

For BNE <1 at $080C:

Search local_labels for n==1 and addr < $080C
Finds $0802
Offset = $0802 - ($080C + 2) = -$0C = -12 bytes backward ✓

BRANCH_MNEMONICS = {'BEQ','BNE','BCC','BCS','BPL','BMI','BVC','BVS'}

# ── Assembler class ───────────────────────────────────────────
class Assembler:
    def __init__(self):
        self.pc            = 0x0800
        self.origin        = 0x0800
        self.output        = bytearray()
        self.global_labels = {}          # name → address
        self.local_labels  = []          # [(number, address), ...]
        self.equates       = {}          # name → value
        self.errors        = []

    # ── Public entry point ────────────────────────────────────
    def assemble(self, source):
        lines = source.splitlines()
        self.pass1(lines)
        if self.errors:
            return None
        self.pass2(lines)
        return self.output

    # ── Pass 1: collect labels ────────────────────────────────
    def pass1(self, lines):
        pc = self.origin
        for lineno, raw in enumerate(lines, 1):
            line = self.strip_comment(raw).strip()
            if not line:
                continue

            # .ORG directive
            if line.upper().startswith('.ORG'):
                pc = self.parse_number(line.split()[1])
                self.origin = pc
                continue

            # .EQU directive  (NAME .EQU value)
            if '.EQU' in line.upper():
                parts = line.split()
                self.equates[parts[0].upper()] = self.parse_number(parts[2])
                continue

            # .DB / .DW — just advance PC
            if line.upper().startswith('.DB'):
                values = line[3:].split(',')
                pc += len(values)
                continue
            if line.upper().startswith('.DW'):
                values = line[3:].split(',')
                pc += len(values) * 2
                continue

            # Local label ^n
            if line.startswith('^'):
                number = int(line[1:].split()[0])
                self.local_labels.append((number, pc))
                continue

            # Global label NAME:
            if line.endswith(':'):
                self.global_labels[line[:-1].upper()] = pc
                continue

            # Instruction — just measure its size
            parts = line.split(None, 1)
            mnemonic = parts[0].upper()
            operand  = parts[1].strip() if len(parts) > 1 else ''
            pc += self.instruction_size(mnemonic, operand)

    # ── Pass 2: emit bytes ────────────────────────────────────
    def pass2(self, lines):
        self.pc = self.origin
        for lineno, raw in enumerate(lines, 1):
            line = self.strip_comment(raw).strip()
            if not line:
                continue

            try:
                self.emit_line(line, lineno)
            except Exception as e:
                self.errors.append(f"Line {lineno}: {e}  [{raw.strip()}]")

    def emit_line(self, line, lineno):
        # Directives
        if line.upper().startswith('.ORG'):
            self.pc = self.parse_number(line.split()[1])
            return
        if '.EQU' in line.upper():
            return
        if line.upper().startswith('.DB'):
            for v in line[3:].split(','):
                self.emit_byte(self.parse_number(v.strip()))
            return
        if line.upper().startswith('.DW'):
            for v in line[3:].split(','):
                self.emit_word(self.parse_number(v.strip()))
            return

        # Labels — no bytes emitted
        if line.startswith('^') or line.endswith(':'):
            return

        # Instruction
        parts    = line.split(None, 1)
        mnemonic = parts[0].upper()
        operand  = parts[1].strip() if len(parts) > 1 else ''
        self.emit_instruction(mnemonic, operand)

    # ── Instruction emitter ───────────────────────────────────
    def emit_instruction(self, mnemonic, operand):
        if mnemonic not in OPCODES:
            raise ValueError(f"Unknown mnemonic: {mnemonic}")

        modes = OPCODES[mnemonic]

        # Implied / accumulator
        if not operand or operand.upper() == 'A':
            self.emit_byte(modes['imp'])
            return

        # Branch instructions
        if mnemonic in BRANCH_MNEMONICS:
            target = self.resolve_operand(operand)
            offset = target - (self.pc + 2)
            if not -128 <= offset <= 127:
                raise ValueError(f"Branch out of range: {offset}")
            self.emit_byte(modes['rel'])
            self.emit_byte(offset & 0xFF)
            return

        # Immediate  #$xx
        if operand.startswith('#'):
            val = self.parse_number(operand[1:])
            self.emit_byte(modes['imm'])
            self.emit_byte(val & 0xFF)
            return

        # Indirect (zp,X)
        if operand.startswith('(') and operand.upper().endswith(',X)'):
            val = self.parse_number(operand[1:-3])
            self.emit_byte(modes['idx'])
            self.emit_byte(val & 0xFF)
            return

        # Indirect (zp),Y
        if operand.startswith('(') and operand.upper().endswith('),Y'):
            val = self.parse_number(operand[1:-3])
            self.emit_byte(modes['idy'])
            self.emit_byte(val & 0xFF)
            return

        # Indirect JMP ($xxxx)
        if operand.startswith('(') and operand.endswith(')'):
            val = self.parse_number(operand[1:-1])
            self.emit_byte(modes['ind'])
            self.emit_word(val)
            return

        # Absolute or zero page, with optional ,X / ,Y
        index = ''
        if ',' in operand:
            operand, index = operand.rsplit(',', 1)
            index = index.strip().upper()

        val = self.resolve_operand(operand.strip())

        if val < 0x100:  # zero page
            if index == 'X':
                self.emit_byte(modes['zpx']); self.emit_byte(val)
            elif index == 'Y':
                self.emit_byte(modes.get('zpy', modes['zp'])); self.emit_byte(val)
            else:
                self.emit_byte(modes['zp']); self.emit_byte(val)
        else:            # absolute
            if index == 'X':
                self.emit_byte(modes['abx']); self.emit_word(val)
            elif index == 'Y':
                self.emit_byte(modes['aby']); self.emit_word(val)
            else:
                self.emit_byte(modes['abs']); self.emit_word(val)

    # ── Local label resolution ────────────────────────────────
    def resolve_operand(self, operand):
        operand = operand.strip()

        if operand.startswith('>'):       # forward local
            return self.find_local(int(operand[1:]), 'forward')

        if operand.startswith('<'):       # backward local
            return self.find_local(int(operand[1:]), 'backward')

        if operand.upper() in self.global_labels:
            return self.global_labels[operand.upper()]

        if operand.upper() in self.equates:
            return self.equates[operand.upper()]

        return self.parse_number(operand)

    def find_local(self, number, direction):
        if direction == 'forward':
            candidates = [a for n,a in self.local_labels
                          if n == number and a > self.pc]
            if not candidates:
                raise ValueError(f"No forward label ^{number} from ${self.pc:04X}")
            return min(candidates)
        else:
            candidates = [a for n,a in self.local_labels
                          if n == number and a < self.pc]
            if not candidates:
                raise ValueError(f"No backward label ^{number} from ${self.pc:04X}")
            return max(candidates)

    # ── Helpers ───────────────────────────────────────────────
    def emit_byte(self, value):
        self.output.append(value & 0xFF)
        self.pc += 1

    def emit_word(self, value):
        self.output.append(value & 0xFF)
        self.output.append((value >> 8) & 0xFF)
        self.pc += 2

    def instruction_size(self, mnemonic, operand):
        if mnemonic not in OPCODES:
            return 0
        if not operand or operand.upper() == 'A':
            return 1
        if mnemonic in BRANCH_MNEMONICS:
            return 2
        if operand.startswith('#'):
            return 2
        if operand.startswith('('):
            return 2 if 'X)' in operand or '),Y' in operand else 3
        val_str = operand.split(',')[0].strip()
        try:
            val = self.parse_number(val_str)
            return 2 if val < 0x100 else 3
        except:
            return 3  # assume absolute if label

    def parse_number(self, s):
        s = s.strip()
        if s.startswith('$'):
            return int(s[1:], 16)
        if s.startswith('%'):
            return int(s[1:], 2)
        if s.startswith("'"):
            return ord(s[1])       # character literal
        return int(s)

    def strip_comment(self, line):
        if ';' in line:
            return line[:line.index(';')]
        return line

# ── CLI ───────────────────────────────────────────────────────
if __name__ == '__main__':
    if len(sys.argv) < 2:
        print("Usage: assembler.py source.asm [-o output.bin]")
        sys.exit(1)

    src_path = Path(sys.argv[1])
    out_path = Path(sys.argv[3]) if '-o' in sys.argv else src_path.with_suffix('.bin')

    source = src_path.read_text()
    asm    = Assembler()
    result = asm.assemble(source)

    if asm.errors:
        for e in asm.errors:
            print(f"ERROR: {e}", file=sys.stderr)
        sys.exit(1)

    out_path.write_bytes(result)
    print(f"OK  {len(result)} bytes → {out_path}")

The Implementation
python
class AssemblerWorkspace:
    """Simulates Apple II memory for multi-part assembly."""
    
    def __init__(self):
        self.memory = bytearray(65536)  # full 64KB
        self.parts  = {}                # name → (start, length)
    
    def bsave(self, name, start, length):
        """Simulate BSAVE — snapshot a memory region to a file."""
        data = bytes(self.memory[start:start + length])
        
        # Save to Linux file as well, preserving the workflow
        path = Path(name)
        path.write_bytes(data)
        
        # Also remember it internally
        self.parts[name] = (start, length)
        print(f"BSAVE {name} A=${start:04X} L=${length:04X} ({length} bytes)")
        return data
    
    def bload(self, name, address=None):
        """Simulate BLOAD — load a file into memory at given address."""
        path = Path(name)
        
        if not path.exists():
            raise FileNotFoundError(f"BLOAD: {name} not found")
        
        data = path.read_bytes()
        
        # If no address given, use original save address
        if address is None and name in self.parts:
            address = self.parts[name][0]
        
        if address is None:
            raise ValueError(f"BLOAD {name}: no address specified")
        
        self.memory[address:address + len(data)] = data
        print(f"BLOAD {name} → ${address:04X} ({len(data)} bytes)")
    
    def assemble_part(self, source_path, expected_org=None):
        """Assemble one source file and drop result into memory."""
        source = Path(source_path).read_text()
        
        asm    = Assembler()
        binary = asm.assemble(source)
        
        if asm.errors:
            for e in asm.errors:
                print(f"ERROR in {source_path}: {e}")
            return False
        
        origin = asm.origin  # .ORG from source file
        
        if expected_org and origin != expected_org:
            print(f"WARNING: {source_path} ORG=${origin:04X} "
                  f"expected=${expected_org:04X}")
        
        # Drop into memory at the address the source declared
        self.memory[origin:origin + len(binary)] = binary
        print(f"Assembled {source_path}: "
              f"${origin:04X}-${origin+len(binary)-1:04X} "
              f"({len(binary)} bytes)")
        return True

def emit_line(self, line, lineno):
    upper = line.upper().strip()
    
    # Intercept BSAVE directive
    # e.g:  BSAVE PART1,A$0800,L$0400

        return
    
    # Intercept BLOAD directive
    # e.g:  BLOAD PART1,A$0800
    if upper.startswith('BLOAD'):
        self.handle_bload(line)
        return
    
    # ... normal assembly continues

def handle_bsave(self, line):
    # Parse:  BSAVE NAME,A$xxxx,L$xxxx
    import re
    m = re.match(
        r'BSAVE\s+(\S+),A\$([0-9A-Fa-f]+),L\$([0-9A-Fa-f]+)',
        line, re.IGNORECASE
    )
    if m:
        name   = m.group(1)
        start  = int(m.group(2), 16)
        length = int(m.group(3), 16)
        self.workspace.bsave(name, start, length)

def handle_bload(self, line):
    # Parse:  BLOAD NAME,A$xxxx  or  BLOAD NAME
    import re
    m = re.match(
        r'BLOAD\s+(\S+)(?:,A\$([0-9A-Fa-f]+))?',
        line, re.IGNORECASE
    )
    if m:
        name    = m.group(1)
        address = int(m.group(2), 16) if m.group(2) else None
        self.workspace.bload(name, address)

python
import time

APPLE2_HZ     = 1_023_000   # target cycles per second
FRAME_CYCLES  = APPLE2_HZ // 60  # ~17050 cycles per frame at 60fps
FRAME_TIME    = 1.0 / 60         # 16.67ms per frame

def run_frame(cpu, memory):
    frame_start = time.perf_counter()
    
    cycles_run = 0
    while cycles_run < FRAME_CYCLES:
        cycles_run += cpu.step(memory)  # returns cycles used
    
    # Throttle — sleep remaining frame time if we ran too fast
    elapsed = time.perf_counter() - frame_start
    remaining = FRAME_TIME - elapsed
    if remaining > 0:
        time.sleep(remaining)
    # If remaining < 0 we're running slow — that's the real problem
    return remaining > 0  # True if keeping up
If Python is fast enough on Zero 2W to finish 17050 cycles before 16.67ms is up, you're golden. If not, you'll notice programs running slower than original — which for most purposes is acceptable for a hobbyist tool, especially if it's only 10-20% slow.

The critical optimisation for the CPU loop is keeping everything in local variables and avoiding dot notation in the hot path:

def run(self, cycles_to_run):
    # Pull everything into locals — faster than self.x in tight loop
    memory  = self._memory
    pc      = self.pc
    sp      = self.sp
    a       = self.a
    x       = self.x
    y       = self.y
    flags   = self.flags
    cycles  = 0
    
    while cycles < cycles_to_run:
        op = memory[pc]
        pc += 1
        
        # Flat if/elif chain is faster than dict dispatch in Python
        if op == 0xA9:    # LDA immediate
            a = memory[pc]; pc += 1
            flags = (flags & 0x7D) | (0x02 if a == 0 else 0) | (a & 0x80)
            cycles += 2
        
        elif op == 0xA5:  # LDA zero page
            a = memory[memory[pc]]; pc += 1
            flags = (flags & 0x7D) | (0x02 if a == 0 else 0) | (a & 0x80)
            cycles += 3
        
        elif op == 0xAD:  # LDA absolute
            addr = memory[pc] | (memory[pc+1] << 8); pc += 2
            a = memory[addr]
            flags = (flags & 0x7D) | (0x02 if a == 0 else 0) | (a & 0x80)
            cycles += 4
        
        # ... all 151 opcodes
    
    # Write back to self
    self.pc = pc
    self.sp = sp
    self.a  = a
    self.x  = x
    self.y  = y
    self.flags = flags
    
    return cycles

Writing to Framebuffer in Python
python
import mmap
import struct
import os

class Framebuffer:
    def __init__(self, device='/dev/fb0'):
        # Get screen info via fbset or ioctl
        self.width  = 1920
        self.height = 1080
        self.bpp    = 4      # 32 bits = 4 bytes per pixel
        
        self.fb   = open(device, 'rb+')
        self.size = self.width * self.height * self.bpp
        self.buf  = mmap.mmap(
            self.fb.fileno(), 
            self.size,
            mmap.MAP_SHARED,
            mmap.PROT_READ | mmap.PROT_WRITE
        )
    
    def put_pixel(self, x, y, r, g, b):
        offset = (y * self.width + x) * self.bpp
        # Pi framebuffer is typically BGRA order
        self.buf[offset:offset+4] = bytes([b, g, r, 0xFF])
    
    def fill_rect(self, x, y, w, h, r, g, b):
        color = bytes([b, g, r, 0xFF])
        row   = color * w
        for row_y in range(h):
            offset = ((y + row_y) * self.width + x) * self.bpp
            self.buf[offset:offset + w * 4] = row
    
    def clear(self, r=0, g=0, b=0):
        self.fill_rect(0, 0, self.width, self.height, r, g, b)
    
    def flush(self):
        self.buf.flush()
Rendering the Apple II Screen
For your emulator the Apple II hi-res screen (280x192) needs to be scaled up to something visible on a modern monitor. A 3x scale gives 840x576 which is comfortable:

python
SCALE       = 3
SCREEN_W    = 280 * SCALE   # 840
SCREEN_H    = 192 * SCALE   # 576
OFFSET_X    = (1920 - SCREEN_W) // 2  # centered horizontally
OFFSET_Y    = (1080 - SCREEN_H) // 2  # centered vertically

# Apple II hi-res color palette (NTSC approximation)
# Each byte has a high bit (palette shift) and 7 pixel bits
# Color depends on bit position (odd/even) and palette bit
COLORS = {
    # (palette_bit, bit_position_odd) → (r, g, b)
    (0, 0): (0,   0,   0),    # black
    (0, 1): (20,  245, 60),   # green
    (1, 0): (255, 68,  253),  # violet  
    (1, 1): (255, 255, 255),  # white
    # with neighbors this gets more complex for full NTSC
}

def render_hires(fb, memory):
    """Render Apple II hi-res page 1 ($2000-$3FFF) to framebuffer."""
    
    for y in range(192):
        base_addr = hires_row_addr(y)  # non-linear Apple II addressing
        
        for byte_x in range(40):
            addr    = 0x2000 + base_addr + byte_x
            byte    = memory[addr]
            palette = (byte >> 7) & 1   # high bit = color palette
            
            for bit in range(7):
                pixel_on = (byte >> bit) & 1
                odd      = (byte_x * 7 + bit) % 2
                
                if pixel_on:
                    r, g, b = COLORS[(palette, odd)]
                else:
                    r, g, b = 0, 0, 0
                
                # Scale up and center on screen
                screen_x = OFFSET_X + (byte_x * 7 + bit) * SCALE
                screen_y = OFFSET_Y + y * SCALE
                
                fb.fill_rect(screen_x, screen_y, SCALE, SCALE, r, g, b)
    
    fb.flush()

def hires_row_addr(y):
    """Apple II hi-res has a non-linear row layout."""
    return ((y & 0x07) << 10) | \
           ((y & 0x38) << 4)  | \
           ((y & 0xC0) * 40 // 64)

Performance Problem With Pure Python Rendering
The naive per-pixel Python loop above is too slow for 60fps. For 280x192 pixels scaled 3x you're setting ~161,000 pixels per frame. In Python that's painfully slow.

The solution is numpy — write the entire frame as an array operation:

python
import numpy as np

class FastFramebuffer:
    def __init__(self):
        self.width  = 1920
        self.height = 1080
        # Frame as numpy array — shape (height, width, 4) BGRA
        self.frame  = np.zeros((self.height, self.width, 4), dtype=np.uint8)
        self.fb     = open('/dev/fb0', 'rb+')
        self.buf    = mmap.mmap(self.fb.fileno(), 
                                self.width * self.height * 4)
    
    def render_hires_fast(self, memory):
        """Render entire hi-res screen using numpy — much faster."""
        
        # Extract all 40x192 bytes from Apple II memory at once
        hires_bytes = np.frombuffer(memory, dtype=np.uint8)
        
        # Build pixel array for 280x192 screen
        pixels = np.zeros((192, 280), dtype=np.uint8)
        
        for y in range(192):
            addr = 0x2000 + hires_row_addr(y)
            for bx in range(40):
                byte    = memory[addr + bx]
                palette = (byte >> 7) & 1
                for bit in range(7):
                    x   = bx * 7 + bit
                    on  = (byte >> bit) & 1
                    pixels[y, x] = on * (1 + palette)  # 0=black,1=green,2=violet,3=white
        
        # Map pixel values to colors
        color_map = np.array([
            [0,   0,   0,   255],  # 0 black
            [20,  245, 60,  255],  # 1 green
            [255, 68,  253, 255],  # 2 violet
            [255, 255, 255, 255],  # 3 white
        ], dtype=np.uint8)
        
        # Color lookup — fast numpy indexing
        colored = color_map[pixels]  # shape (192, 280, 4)
        
        # Scale up with numpy repeat
        scaled = np.repeat(np.repeat(colored, SCALE, axis=0), SCALE, axis=1)
        
        # Place in center of frame
        self.frame[OFFSET_Y:OFFSET_Y+SCREEN_H, 
                   OFFSET_X:OFFSET_X+SCREEN_W] = scaled
        
        # Write to framebuffer in one shot
        self.buf.seek(0)
        self.buf.write(self.frame.tobytes())
This renders the entire Apple II screen in a few milliseconds rather than hundreds — numpy pushes the pixel work into C under the hood.

CPU core — the hot loop:

c
// cpu6502.h
typedef struct {
    uint16_t pc;
    uint8_t  sp, a, x, y, flags;
    uint8_t  memory[65536];
    int      cycles;
} CPU;

// Flags
#define FLAG_C  0x01
#define FLAG_Z  0x02
#define FLAG_I  0x04
#define FLAG_D  0x08
#define FLAG_B  0x10
#define FLAG_V  0x40
#define FLAG_N  0x80

#define SET_NZ(val) \
    cpu->flags = (cpu->flags & ~(FLAG_N|FLAG_Z)) | \
                 ((val) & 0x80) | \
                 ((val) == 0 ? FLAG_Z : 0)

void cpu_step(CPU *cpu) {
    uint8_t op = cpu->memory[cpu->pc++];
    
    switch(op) {
        case 0xA9:  // LDA immediate
            cpu->a = cpu->memory[cpu->pc++];
            SET_NZ(cpu->a);
            cpu->cycles += 2;
            break;
        
        case 0xA5:  // LDA zero page
            cpu->a = cpu->memory[cpu->memory[cpu->pc++]];
            SET_NZ(cpu->a);
            cpu->cycles += 3;
            break;
        
        case 0xAD: {// LDA absolute
            uint16_t addr = cpu->memory[cpu->pc] |
                           (cpu->memory[cpu->pc+1] << 8);
            cpu->pc += 2;
            cpu->a   = cpu->memory[addr];
            SET_NZ(cpu->a);
            cpu->cycles += 4;
            break;
        }
        
        case 0x8D: {// STA absolute
            uint16_t addr = cpu->memory[cpu->pc] |
                           (cpu->memory[cpu->pc+1] << 8);
            cpu->pc += 2;
            cpu_write(cpu, addr, cpu->a);  // write checks soft switches
            cpu->cycles += 4;
            break;
        }
        
        case 0x4C: {// JMP absolute
            cpu->pc = cpu->memory[cpu->pc] |
                     (cpu->memory[cpu->pc+1] << 8);
            cpu->cycles += 3;
            break;
        }
        
        case 0x20: {// JSR absolute
            uint16_t addr = cpu->memory[cpu->pc] |
                           (cpu->memory[cpu->pc+1] << 8);
            uint16_t ret  = cpu->pc + 1;
            cpu->memory[0x100 + cpu->sp--] = (ret >> 8) & 0xFF;
            cpu->memory[0x100 + cpu->sp--] =  ret       & 0xFF;
            cpu->pc     = addr;
            cpu->cycles += 6;
            break;
        }
        
        case 0x60: {// RTS
            uint16_t addr = cpu->memory[0x100 + ++cpu->sp];
            addr |= cpu->memory[0x100 + ++cpu->sp] << 8;
            cpu->pc     = addr + 1;
            cpu->cycles += 6;
            break;
        }
        
        case 0xD0: {// BNE
            int8_t offset = (int8_t)cpu->memory[cpu->pc++];
            if (!(cpu->flags & FLAG_Z)) {
                cpu->pc    += offset;
                cpu->cycles += 3;
            } else {
                cpu->cycles += 2;
            }
            break;
        }
        
        // ... all opcodes
    }
}
HLE intercept — check before each step:

c
// Known ROM vectors to intercept
#define COUT    0xFDED
#define RDKEY   0xFD0C
#define DOS_ENTRY 0x03D9

int check_hle(CPU *cpu, Emulator *emu) {
    switch(cpu->pc) {
        case COUT:
            hle_cout(cpu, emu);
            cpu_rts(cpu);  // simulate RTS
            return 1;
        
        case RDKEY:
            hle_rdkey(cpu, emu);
            cpu_rts(cpu);
            return 1;
        
        case DOS_ENTRY:
            if (cpu->memory[0x0243] == 2) {  // drive 2?
                hle_dos(cpu, emu);
                cpu_rts(cpu);
                return 1;
            }
            return 0;
    }
    return 0;
}

void emulator_run_frame(CPU *cpu, Emulator *emu) {
    int cycles_per_frame = 1023000 / 60;
    int cycles = 0;
    
    while (cycles < cycles_per_frame) {
        if (!check_hle(cpu, emu)) {
            cpu_step(cpu);
        }
        cycles = cpu->cycles;
    }
    cpu->cycles = 0;
}
Framebuffer rendering:

c
#include <fcntl.h>
#include <sys/mman.h>
#include <sys/ioctl.h>
#include <linux/fb.h>

typedef struct {
    int      fd;
    uint32_t *pixels;
    int      width, height;
} Framebuffer;

Framebuffer fb_open(const char *device) {
    Framebuffer fb;
    struct fb_var_screeninfo vinfo;
    
    fb.fd = open(device, O_RDWR);
    ioctl(fb.fd, FBIOGET_VSCREENINFO, &vinfo);
    
    fb.width  = vinfo.xres;
    fb.height = vinfo.yres;
    
    size_t size = fb.width * fb.height * 4;
    fb.pixels   = mmap(0, size,
                       PROT_READ|PROT_WRITE,
                       MAP_SHARED, fb.fd, 0);
    return fb;
}

// Apple II hi-res row address (non-linear layout)
int hires_row_addr(int y) {
    return ((y & 0x07) << 10) |
           ((y & 0x38) <<  4) |
           ((y & 0xC0) * 40 / 64);
}

// NTSC color palette for hi-res
uint32_t HIRES_COLORS[2][2] = {
    //  even pixel        odd pixel
    { 0xFF000000,       0xFF000000 },  // palette 0 off: black
    { 0xFF3CF51C,       0xFFFF44FD },  // palette 0 on:  green / violet
};

void render_hires(Framebuffer *fb, uint8_t *memory) {
    int scale    = 3;
    int offset_x = (fb->width  - 280 * scale) / 2;
    int offset_y = (fb->height - 192 * scale) / 2;
    
    for (int y = 0; y < 192; y++) {
        int base = 0x2000 + hires_row_addr(y);
        
        for (int bx = 0; bx < 40; bx++) {
            uint8_t byte    = memory[base + bx];
            int     palette = (byte >> 7) & 1;
            
            for (int bit = 0; bit < 7; bit++) {
                int      on    = (byte >> bit) & 1;
                int      odd   = (bx * 7 + bit) & 1;
                uint32_t color = on
                    ? HIRES_COLORS[palette][odd]
                    : 0xFF000000;
                
                // Scale up
                int sx = offset_x + (bx * 7 + bit) * scale;
                int sy = offset_y + y * scale;
                
                for (int dy = 0; dy < scale; dy++)
                    for (int dx = 0; dx < scale; dx++)
                        fb->pixels[(sy+dy)*fb->width + (sx+dx)] = color;
            }
        }
    }
}

c
void hle_dos(CPU *cpu, Emulator *emu) {
    uint8_t  command  = cpu->memory[0x0242];
    char    *filename = (char*)&cpu->memory[0x0200];
    char     path[512];
    
    snprintf(path, sizeof(path), "%s/%s", emu->drive2_path, filename);
    
    switch(command) {
        case 0x04: {  // SAVE / BSAVE
            uint16_t start  = cpu->memory[0x026C] |
                             (cpu->memory[0x026D] << 8);
            uint16_t length = cpu->memory[0x026E] |
                             (cpu->memory[0x026F] << 8);
            FILE *f = fopen(path, "wb");
            if (f) {
                fwrite(&cpu->memory[start], 1, length, f);
                fclose(f);
            }
            break;
        }
        case 0x02: {  // LOAD / BLOAD
            uint16_t addr = cpu->memory[0x026C] |
                           (cpu->memory[0x026D] << 8);
            FILE *f = fopen(path, "rb");
            if (f) {
                fread(&cpu->memory[addr], 1, 65536 - addr, f);
                fclose(f);
            }
            break;
        }
    }
    cpu->a = 0;  // no error
}

#include <time.h>

int main(int argc, char *argv[]) {
    CPU        cpu = {0};
    Emulator   emu = {0};
    Framebuffer fb = fb_open("/dev/fb0");
    
    // Load .dsk image for drive 1
    load_dsk(&cpu, argv[1]);
    
    // Set drive 2 to Linux folder
    emu.drive2_path = argv[2];
    
    // Patch HLE entry points
    patch_hle(&cpu);
    
    // Reset vector
    cpu.pc = cpu.memory[0xFFFC] | (cpu.memory[0xFFFD] << 8);
    
    keyboard_init();
    
    // Main loop — 60fps
    struct timespec frame_time = { .tv_nsec = 16666667 }; // 1/60 sec
    
    while (1) {
        struct timespec start, end;
        clock_gettime(CLOCK_MONOTONIC, &start);
        
        // Run one frame of CPU
        emulator_run_frame(&cpu, &emu);
        
        // Render to framebuffer
        render_hires(&fb, cpu.memory);
        
        // Handle keyboard
        uint8_t key;
        if (keyboard_read(&key))
            cpu.memory[0xC000] = key | 0x80;  // Apple II keyboard strobe
        
        // Sleep remaining frame time
        clock_gettime(CLOCK_MONOTONIC, &end);
        long elapsed = (end.tv_sec  - start.tv_sec)  * 1000000000L +
                       (end.tv_nsec - start.tv_nsec);
        long remaining = 16666667L - elapsed;
        if (remaining > 0) {
            struct timespec sleep = { .tv_nsec = remaining };
            nanosleep(&sleep, NULL);
        }
    }
}

Connecting the Apple II Keyboard
The Apple II keyboard is a straightforward matrix — typically 8 rows × 8 columns of switches plus a few special lines. The encoder chip on the original board converts the matrix to ASCII, but you can bypass it entirely and let the Pi read the raw matrix.

Option A — Keep the original encoder chip:

The Apple II encoder outputs a strobe signal and a 7-bit ASCII code. You can read this directly from Pi GPIO with very simple code:

c
// Encoder outputs ASCII on 7 data pins + 1 strobe pin
#define STROBE_PIN  4
#define DATA_PINS   {17, 27, 22, 5, 6, 13, 19}  // 7 bits

int read_apple2_key() {
    if (!gpio_read(STROBE_PIN))
        return -1;  // no key
    
    int key = 0;
    int pins[] = DATA_PINS;
    for (int i = 0; i < 7; i++)
        key |= gpio_read(pins[i]) << i;
    
    return key & 0x7F;  // ASCII value
}
This is the easiest approach — the encoder chip does all the work and you just read the result.

Option B — Bypass encoder, read matrix directly:

More work but gives you full control, including modifier keys the encoder might not handle the way you want:

c
#define ROWS 8
#define COLS 8

int row_pins[] = {4,  17, 27, 22, 5,  6,  13, 19};
int col_pins[] = {12, 16, 20, 21, 26, 24, 25,  8};

// Keymap[row][col] = ASCII
uint8_t keymap[8][8] = {
    {'3','4','5','6','7','8','9','0'},
    {'E','R','T','Y','U','I','O','P'},
    {'D','F','G','H','J','K','L',';'},
    {'C','V','B','N','M',',','.', '/'},
    // ... etc
};

int scan_keyboard() {
    for (int r = 0; r < ROWS; r++) {
        gpio_write(row_pins[r], LOW);
        for (int c = 0; c < COLS; c++) {
            if (gpio_read(col_pins[c]) == LOW)
                return keymap[r][c];
        }
        gpio_write(row_pins[r], HIGH);
    }
    return -1;
}

Critical Issue — Voltage
This is the most important thing to get right. The Apple II keyboard runs at 5V logic. The Pi Zero GPIO pins are 3.3V only — feeding 5V into a Pi GPIO pin will damage it.

You need level shifting on the protoboard:

Keyboard (5V) ──→ Level Shifter ──→ Pi GPIO (3.3V)
A cheap 74LVC245 or BSS138-based bidirectional level shifter module handles this. Or a simple resistor voltage divider for input-only signals:

Keyboard 5V signal
        │
       4.7kΩ
        │
        ├──────── Pi GPIO pin (3.3V safe)
        │
       10kΩ
        │
       GND
Output = 5V × 10/(4.7+10) = 3.4V → close enough for 3.3V logic ✓

        gpiod_line_request_input(data[i], "apple2");

}

        return -1;

The C Code For This
Clean and simple since the encoder does all the hard work:

c
#include <gpiod.h>
#include <string.h>

#define STROBE_GPIO   4
#define RESET_GPIO   17
int data_gpios[] = {27, 22, 5, 6, 13, 19, 26};  // D0-D6

struct gpiod_chip *chip;
struct gpiod_line *strobe_line;
struct gpiod_line *reset_line;
struct gpiod_line *data_lines[7];

void keyboard_init() {
    chip         = gpiod_chip_open("/dev/gpiochip0");
    strobe_line  = gpiod_chip_get_line(chip, STROBE_GPIO);
    reset_line   = gpiod_chip_get_line(chip, RESET_GPIO);
    
    gpiod_line_request_input(strobe_line, "apple2_kbd");
    gpiod_line_request_input(reset_line,  "apple2_kbd");
    
    for (int i = 0; i < 7; i++) {
        data_lines[i] = gpiod_chip_get_line(chip, data_gpios[i]);
        gpiod_line_request_input(data_lines[i], "apple2_kbd");
    }
}

typedef struct {
    int     key;      // ASCII value, -1 if no key
    int     reset;    // 1 if CTRL+RESET pressed
} KeyEvent;

KeyEvent keyboard_read() {
    KeyEvent ev = { -1, 0 };
    
    // Check reset first (active low — grounded when pressed)
    if (!gpiod_line_get_value(reset_line))
        ev.reset = 1;
    
    // Strobe high means a key is ready
    if (!gpiod_line_get_value(strobe_line))
        return ev;
    
    // Read 7-bit ASCII from data lines
    int ascii = 0;
    for (int i = 0; i < 7; i++)
        ascii |= gpiod_line_get_value(data_lines[i]) << i;
    
    ev.key = ascii & 0x7F;
    return ev;
}

// In your main emulator loop:
void handle_keyboard(CPU *cpu) {
    KeyEvent ev = keyboard_read();
    
    if (ev.reset) {
        cpu_reset(cpu);
        return;
    }
    
    if (ev.key >= 0) {
        // Apple II reads keyboard at $C000
        // High bit set = key available
        cpu->memory[0xC000] = ev.key | 0x80;
        // Clear strobe at $C010
        // (emulator clears it when $C010 is read)
    }
}

Option A — Use the original motherboard connector

Red     → +5V
Yellow  → +12V
Blue    → -12V
White   → -5V  (may not be present on all units)
Black   → GND
Orange  → +5V standby (if present)
Solder or use a Wago connector to tap +5V (red), -12V (blue), and GND (black) and run them to your protoboard.

Option C — Load resistor on unused rails

Some older PSUs need a minimum load on their rails to regulate properly. If you're only powering the keyboard and Pi you might find +5V regulates fine but -12V drifts without the motherboard load.

Easy fix — put a small resistor across -12V to GND to provide a dummy load:

-12V ──→ 560Ω resistor ──→ GND

Power dissipated: 12V²/560Ω = ~0.26W  → use 0.5W resistor
Current drawn:    12V/560Ω  = ~21mA   → enough to stabilize rail
For +5V the Pi itself (200-500mA) is more than enough load.

The Pi Zero 2W should be powered from its USB input or the 5V GPIO pin, not from the Unitron PSU +5V directly — unless you verify the PSU +5V is clean and stable enough.

The safest approach:

Unitron PSU +5V → keyboard only + 74LVC245 Vcc A side
USB phone charger → Pi Zero 2W USB power input
Keeping Pi power separate from the vintage PSU avoids any noise or regulation issues from the old supply affecting the Pi. The PSU powers the keyboard exactly as it always did, the Pi has its own clean modern power source.

If you want to power everything from one source, add a small modern 5V regulator (like an LM2596 module) between the Unitron +5V and the Pi, giving the Pi clean regulated power even if the old PSU has some ripple.

Do you recommend optical chip isolator to keep raspberry electrically isolated from the keyboard? Or is it overkill?

Feb 28
Good instinct to think about it. Let me give you an honest assessment.

What An Optoisolator Does
Keyboard side (5V/-12V)          Pi side (3.3V)
                    ┌─────────────┐
Signal ──→ LED ──→  │   optical   │ ──→ Phototransistor ──→ GPIO
                    │   barrier   │
GND (keyboard) ──X──│─────────────│──X── GND (Pi)
                    └─────────────┘
                    No electrical
                    connection at all
Grounds are completely separate. A fault on the keyboard side cannot reach the Pi.

The Real Risks Without Isolation
To decide if it's overkill you need to know what could actually go wrong:

Risk 1 — -12V accidentally reaching Pi GPIO This is the real danger in your setup. The keyboard connector has -12V on pin 15 sitting right next to signal pins. A wiring mistake, a solder bridge, or a connector inserted slightly off could put -12V onto a Pi GPIO. That kills the Pi instantly. The level shifter alone won't protect against this.

Risk 2 — PSU noise on keyboard signals Old switching PSUs generate noise. This noise could couple into your GPIO signals and cause phantom keypresses or corrupt reads. Optoisolators naturally filter this out.

Risk 3 — Ground loops If keyboard ground and Pi ground are connected but they're powered from different sources (Unitron PSU + USB charger), you can get ground loop currents that cause erratic behavior or hum.

Risk 4 — Static discharge from typing Someone typing vigorously on a 40-year-old keyboard could generate static that travels through the matrix into your signal lines. Unlikely but possible.

The Case FOR Optoisolators Here
A few things specific to your build make isolation more attractive than in a typical project:

-12V is present on the same connector — this is the strongest argument. Unlike a pure 5V system where the worst a fault can do is overvoltage a GPIO slightly, you have an actual negative voltage rail that would cause serious damage if it ever reached the Pi. The 16-pin connector with -12V on pin 15 right next to signals is a real hazard.

It's a 40-year-old keyboard — the original wiring and connector may have deteriorated. Insulation on wires gets brittle, connectors get oxidized and can make unexpected contact. You don't fully know the condition of every wire inside that keyboard cable.

You only need 9 channels — optoisolators are cheap. A PC817 4-channel opto chip costs almost nothing. For 9 signals you need 3 chips — maybe $2 total.

It's a one-time build — you're not mass producing this. The extra 20 minutes of soldering is worth the peace of mind.

The Pi Zero 2W is the heart of the project — if a wiring mistake kills it, you're ordering another one and rebuilding. The optoisolator is cheaper than that risk.

The Case AGAINST (Why It Might Be Overkill)
Signal timing — the AY-5-3600 strobe and data signals aren't particularly fast, but optoisolators add propagation delay (typically 5-20 microseconds). For keyboard scanning this is completely irrelevant — human typing is milliseconds — so this concern doesn't apply here at all.

With careful wiring it's fine — plenty of people interface 5V vintage keyboards to 3.3V microcontrollers with just a level shifter and no problems. If your wiring is clean and you're confident pin 15 is isolated properly, the risk is low.

Complexity — adds more components to the protoboard.

My Honest Recommendation
Do it, but targeted isolation rather than full isolation.

The specific thing you're protecting against is -12V reaching the Pi. So:

Protect with optoisolator:    Strobe, Data 0-6 (the 8 signal lines)
Protect differently:          ~Reset (see below)
Isolate physically:           Pin 15 -12V (don't run it near Pi at all)
For the Reset line specifically, use a simple RC filter plus Schottky diode clamp rather than an opto — it's simpler and the reset signal is slow:

~Reset ──→ 10kΩ ──→ GPIO
                ↓
               100nF
                ↓
               GND
           + BAT54 Schottky from GPIO to 3.3V rail
           (clamps any overvoltage to 3.3V + 0.3V = 3.6V max)
Practical Optoisolator Circuit
The PC817 is cheap, common, and perfect for this:

Keyboard side (5V)              Pi side (3.3V)
                    PC817
+5V ──→ 270Ω ──→ Anode
                 Cathode ──→ Signal (strobe/data)
                 
                          Collector ──→ 3.3V (pull up)
                          Emitter   ──→ GPIO pin
                                    ──→ 10kΩ to GND

One PC817 per signal line
Need 3x PC817 quad packages for 8 signals + strobe
The 270Ω resistor limits LED current:

(5V - 1.2V forward voltage) / 270Ω = ~14mA  ✓ within spec
Pull-up on Pi side:

When LED off (no signal):  GPIO pulled high via 10kΩ = logic 1
When LED on (signal):      Transistor pulls GPIO low = logic 0
Note: signal is inverted — handle in software with a simple NOT

Bottom Line
Not overkill given the -12V present in the connector. The PC817 costs almost nothing, the circuit is simple, and it means a wiring mistake on the keyboard side cannot damage the Pi. For a one-off hobbyist build that you want to be reliable and show off to people, the extra protection is worth it. You'd feel worse replacing a Pi Zero 2W than spending an extra $2 on opto chips.

Build Order
Doing it in stages makes troubleshooting easier:

Stage 2 — keyboard section. Connect keyboard connector, measure that the keyboard gets its power. With an oscilloscope or logic analyzer verify strobe and data signals appear when keys are pressed — before connecting to Pi.

Stage 3 — optoisolator section. Wire up PC817s, verify output signals on Pi side are clean 3.3V logic with multimeter.

Stage 4 — Pi section. Only now plug in the Pi. Connect GPIO pins. Test keyboard reading with a simple Python or C test program.

Stage 5 — full emulator. Load the C emulator, boot GraFORTH, type on the real keyboard and see it appear on screen.

USB Keyboard Mapping
One thing worth thinking about — the Apple II only had uppercase letters originally, plus a specific set of control characters. A USB keyboard needs mapping to Apple II expectations:

// Linux keycode → Apple II ASCII
int linux_to_apple2(int keycode, int shift, int ctrl) {
    if (ctrl) {
        // CTRL+key → control characters
        return keycode_to_ascii(keycode) & 0x1F;
    }
    
    switch (keycode) {
        case KEY_ENTER:     return 0x0D;  // CR
        case KEY_BACKSPACE: return 0x08;  // backspace
        case KEY_ESCAPE:    return 0x1B;  // ESC
        case KEY_LEFT:      return 0x08;  // Apple II left arrow
        case KEY_RIGHT:     return 0x15;  // Apple II right arrow
        // Letters — Apple II original was uppercase only
        case KEY_A ... KEY_Z:
            return shift 
                ? (keycode - KEY_A + 'A')   // uppercase
                : (keycode - KEY_A + 'A');   // Apple II default uppercase
        default:
            return keycode_to_ascii(keycode);
    }
}

Sound: apple2 can only send 1 or 0 to the speaker and we control frequency by alternating 1/0 at the speed we want, controlling by software. Is it possible to monitor this port in my emulator and produce the equivalent sound to my HDMI output?

Feb 28
Yes, absolutely possible and it's a fascinating challenge. The Apple II speaker is one of the most primitive audio systems ever put in a computer — and yet people made remarkable music with it.

How The Apple II Speaker Works
Reading or writing memory address $C030 toggles the speaker cone — that's it. No volume control, no waveform, no DAC. Just a physical click:

$C030 access → speaker toggles position (in or out)
$C030 access → speaker toggles back
Frequency is entirely controlled by how fast software accesses $C030:

Toggle every 1000 cycles  → low note  (~500 Hz)
Toggle every 500 cycles   → higher note (~1000 Hz)
Toggle every 113 cycles   → ~4500 Hz (near top of range)
Your emulator already counts CPU cycles for timing — so you have everything you need.

Monitoring The Port In Your Emulator
In your memory bus code, $C030 is a soft switch. Every time it's accessed you record the current cycle count:

c
// In your memory read/write handler
#define SPEAKER_ADDR 0xC030

// Circular buffer of toggle timestamps
#define TOGGLE_BUFFER_SIZE 4096

typedef struct {
    uint64_t toggles[TOGGLE_BUFFER_SIZE];
    int      write_pos;
    int      read_pos;
    uint64_t total_cycles;
} SpeakerState;

SpeakerState speaker = {0};

void speaker_toggle(uint64_t cycle) {
    // Record when this toggle happened
    speaker.toggles[speaker.write_pos % TOGGLE_BUFFER_SIZE] = cycle;
    speaker.write_pos++;
}

// Called from memory bus when $C030 is accessed
uint8_t cpu_read(CPU *cpu, uint16_t addr) {
    if (addr == SPEAKER_ADDR) {
        speaker_toggle(cpu->total_cycles);
        return 0;  // data value doesn't matter
    }
    return cpu->memory[addr];
}

void cpu_write(CPU *cpu, uint16_t addr, uint8_t val) {
    if (addr == SPEAKER_ADDR) {
        speaker_toggle(cpu->total_cycles);
        return;
    }
    cpu->memory[addr] = val;
}
Converting Toggles To Audio
Once per audio frame (say every 1/60th of a second) you convert the recorded toggles into a PCM audio buffer. The speaker is either in the up or down position — you synthesize a square wave from the toggle times:

c
#define SAMPLE_RATE     44100
#define APPLE2_HZ     1023000
#define SAMPLES_PER_FRAME (SAMPLE_RATE / 60)  // 735 samples

// Samples per CPU cycle
#define CYCLES_PER_SAMPLE ((double)APPLE2_HZ / SAMPLE_RATE)

void render_speaker_audio(SpeakerState *spk,
                          int16_t *buffer,
                          int num_samples,
                          uint64_t frame_start_cycle,
                          uint64_t frame_end_cycle) {
    
    int   speaker_pos = 0;  // current speaker position +1 or -1
    int   toggle_idx  = spk->read_pos;
    
    for (int s = 0; s < num_samples; s++) {
        // What cycle does this sample correspond to?
        uint64_t sample_cycle = frame_start_cycle +
            (uint64_t)(s * CYCLES_PER_SAMPLE);
        
        // Apply any toggles that happened before this sample
        while (toggle_idx < spk->write_pos) {
            uint64_t toggle_cycle = 
                spk->toggles[toggle_idx % TOGGLE_BUFFER_SIZE];
            
            if (toggle_cycle > sample_cycle)
                break;
            
            speaker_pos = -speaker_pos;  // flip
            if (speaker_pos == 0) speaker_pos = 1;  // init
            toggle_idx++;
        }
        
        // Output sample — scale to 16-bit audio range
        // Keep amplitude modest to avoid harsh sound
        buffer[s] = speaker_pos * 8000;
    }
    
    spk->read_pos = toggle_idx;
}

int main() {

The Sound Quality Reality
The Apple II speaker produces square waves — inherently harsh and buzzy. Some things to soften it:

Low pass filter in software — real speakers have mechanical inertia that softens the clicks. You can simulate this:

c
void lowpass_filter(int16_t *buffer, int num_samples) {
    static int16_t prev = 0;
    float alpha = 0.3f;  // cutoff — lower = more filtering
    
    for (int i = 0; i < num_samples; i++) {
        buffer[i] = (int16_t)(alpha * buffer[i] + 
                              (1.0f - alpha) * prev);
        prev = buffer[i];
    }
}
Volume control — scale the 8000 amplitude value. Real Apple II was quite quiet.

Beware of silence — when $C030 is never toggled the speaker sits in one position and produces silence correctly. Make sure your buffer outputs zeros in that case, not a DC offset.

What You'll Actually Hear
GraFORTH programs that use sound access $C030 in tight loops — your emulator captures every single toggle because it happens inside the CPU execute loop. The result is faithful square wave reproduction of whatever the original program was doing.

Simple beeps, the classic Apple II boot sound, game sound effects — all come through correctly. The cycle-accurate toggle recording is actually the right way to do this and it's what serious Apple II emulators like AppleWin and LinApple do under the hood.

The HDMI output will carry the sound to whatever monitor or TV you connect, so your Unitron case has authentic Apple II sound coming out of the TV speaker — completing the experience nicely.

The Concept
Emulator detects $C030 access

Why This Is Actually The Most Accurate Approach
Every other approach converts the toggle stream to digital audio and back to analog — introducing latency and approximation. This approach has none of that:

Original Apple II:
6502 → $C030 → speaker driver circuit → speaker cone

Your emulator:
CPU loop → $C030 detected → GPIO toggle → speaker cone
The speaker responds to the actual toggle timing of your emulated CPU. If your emulation timing is accurate, the sound is accurate. It's the same fundamental mechanism.

The Circuit On Your Protoboard
The Unitron speaker is almost certainly a small 8Ω speaker driven by a simple transistor circuit on the original motherboard. You need to replicate just that driver stage:

Pi GPIO 18 ──→ 1kΩ ──→ Base (NPN transistor e.g. 2N2222)
                        Collector ──→ Speaker + ──→ +5V
                        Emitter   ──→ Speaker - ──→ GND

Also add flyback diode across speaker:
+5V ──→ Cathode | Diode | Anode ──→ Collector
(protects transistor from inductive spike when speaker toggles)
The 2N2222 is in every hobbyist parts bin, costs nothing, and handles the speaker current easily.

Or even simpler — if the original speaker is high impedance enough:

GPIO 18 ──→ 270Ω ──→ Speaker + 
GND     ──────────→ Speaker -
GPIO pins can source about 16mA — enough to drive a small speaker directly with a series resistor. Test with multimeter first to check speaker impedance.

The Emulator Code
This is the beautiful part — it's almost nothing:

c
#include <gpiod.h>

#define SPEAKER_GPIO 18

struct gpiod_chip *chip;
struct gpiod_line *speaker_line;
int speaker_state = 0;

void speaker_init() {
    chip         = gpiod_chip_open("/dev/gpiochip0");
    speaker_line = gpiod_chip_get_line(chip, SPEAKER_GPIO);
    gpiod_line_request_output(speaker_line, "apple2_spk", 0);
}

void speaker_toggle() {
    speaker_state ^= 1;  // flip between 0 and 1
    gpiod_line_set_value(speaker_line, speaker_state);
}

// In your memory bus — already where keyboard lives:
uint8_t cpu_read(CPU *cpu, uint16_t addr) {
    switch(addr) {
        case 0xC030:
            speaker_toggle();  // one line
            return 0;
        
        case 0xC000:
            return cpu->keyboard_latch;
        
        // soft switches etc
    }
    return cpu->memory[addr];
}
That's the entire audio system. One function, one GPIO line, toggled whenever $C030 is accessed.

Timing Consideration
The one subtle thing — your emulator runs a frame at a time (17050 cycles per frame). Within that frame the CPU toggles $C030 at whatever rate the program dictates. Those toggles need to happen in real time, not all at once at the end of the frame.

The good news is your CPU loop already runs in real time with a sleep at the end of each frame to hit 60fps. The toggles happen naturally as the CPU executes:

c
while (cpu.total_cycles < frame_end) {
    cpu_step(&cpu);
    // speaker_toggle() fires inside cpu_read()
    // exactly when the instruction executes
    // GPIO changes immediately — real time
}
```

No buffering needed. The GPIO toggles happen at the same moment the emulated instruction executes — which is as close to authentic as you can get without running on real hardware.
```

void dos_prompt(EmulatorState *emu) {
    
    screen_print(emu, "\nDOS VERSION 3.3 SYSTEM MASTER\n");
    screen_print(emu, "APPLE II\n\n");
    
    char input[64];

void cmd_catalog(EmulatorState *emu) {
    screen_print(emu, "\nDISK VOLUME 254\n\n");
    
    // Drive 2 files from Linux folder
    DIR *d = opendir(emu->drive2_path);
    struct dirent *entry;
    while ((entry = readdir(d)) != NULL) {
        if (entry->d_name[0] == '.') continue;
        
        // Get file size in 256-byte Apple "sectors"
        struct stat st;
        stat(entry->d_name, &st);
        int sectors = (st.st_size / 256) + 1;
        
        // Guess type: B for binary, A for text
        char type = is_text_file(entry->d_name) ? 'A' : 'B';
        
        screen_printf(emu, " %c %03d %s\n", type, sectors, entry->d_name);
    }
    closedir(d);
}
BRUN — load binary and execute from 6502 CPU:

c
void cmd_brun(EmulatorState *emu, char *filename) {
    char path[256];
    snprintf(path, sizeof(path), "%s/%s", emu->drive2_path, filename);
    
    FILE *f = fopen(path, "rb");
    if (!f) {
        screen_printf(emu, "\nFILE NOT FOUND");
        return;
    }
    
    // Apple binary format: 4-byte header (address lo/hi, length lo/hi)
    uint16_t load_addr, length;
    fread(&load_addr, 2, 1, f);
    fread(&length,    2, 1, f);
    fread(&emu->cpu.memory[load_addr], 1, length, f);
    fclose(f);
    
    // Jump CPU to load address
    emu->cpu.pc = load_addr;
    cpu_run(emu);   // runs until RTS or BRK
    
    // Return to DOS prompt when program exits
}

void cmd_bsave(EmulatorState *emu, char *args) {
    // Parse: BSAVE FILENAME,A$0800,L$0200
    char filename[32];
    uint16_t addr, length;
    parse_bsave_args(args, filename, &addr, &length);
    
    char path[256];
    snprintf(path, sizeof(path), "%s/%s", emu->drive2_path, filename);
    
    FILE *f = fopen(path, "wb");
    // Write 4-byte Apple binary header
    fwrite(&addr,   2, 1, f);
    fwrite(&length, 2, 1, f);
    // Write data
    fwrite(&emu->cpu.memory[addr], 1, length, f);
    fclose(f);
    
    screen_printf(emu, "\n");  // DOS just prints nothing on success
}
